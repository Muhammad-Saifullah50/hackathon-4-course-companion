# Implementation Plan: AI Course Mentor

**Branch**: `007-ai-course-mentor` | **Date**: 2026-06-16 | **Spec**: [spec.md](./spec.md)
**Input**: Feature specification from `/specs/007-ai-course-mentor/spec.md`

## Summary

Build the AI Course Mentor first for the web app as a Pro-only premium hybrid feature. Add FastAPI premium mentor endpoints, including a ChatKit Python endpoint, guard every endpoint with authenticated server-side Pro access checks, persist learner-owned threads and messages in Neon PostgreSQL, chunk and embed R2 course content into Neon pgvector, and run an OpenAI Agents SDK mentor agent with LiteLLM configured for the environment-specified OpenRouter/Nvidia/Nemotron model. Add a floating ChatKit React chat bubble and pop-up window to the web app shell; do not add a standalone `/mentor` page or Next.js API proxy routes.

## Technical Context

**Language/Version**: Python 3.12+ backend; TypeScript 5.7 with Next.js 16.2.9 and React 19 web frontend  
**Primary Dependencies**: FastAPI, Pydantic v2, SQLAlchemy 2.0 async, asyncpg, pgvector, boto3/R2 content service, Stytch JWT auth, `openai-agents`, `openai-chatkit`, `litellm`, OpenRouter-hosted Nvidia Nemotron model via environment configuration, Next.js App Router, React 19, Tailwind CSS, lucide-react, `@openai/chatkit-react`  
**Storage**: Neon PostgreSQL for mentor threads, mentor messages, daily usage counters, and vectorized course chunks via pgvector; Cloudflare R2 for authoritative course content  
**Testing**: pytest and pytest-asyncio for backend unit/integration tests; Vitest and Testing Library for web UI/API helper tests  
**Target Platform**: Vercel-hosted FastAPI backend and Vercel-hosted Next.js web app  
**Project Type**: Web application with shared FastAPI backend and separate Next.js frontend  
**Performance Goals**: Initial grounded mentor response returns in under 30 seconds in the typical case; list/get thread endpoints return in under 1 second for normal user history sizes; daily quota checks execute in one indexed database lookup/update; mentor turns perform vector retrieval against already-indexed chunks and do not rechunk or re-embed course content during chat  
**Constraints**: Pro-only; user-initiated LLM calls only; maximum five valid Pro learner text messages per UTC day; text input only; no file/image/audio/video input; no course progress, quiz score, or access-tier side effects; no ChatGPT app support in this release; guardrails enforce scope and grounding; no standalone web `/mentor` route; no frontend API proxy routes
**Scale/Scope**: One hybrid feature in this release; per-user thread history retained until account deletion; full history displayed but only most recent 20 thread messages used as response context

**Structure Decision**: Use the existing shared FastAPI backend plus Next.js web app structure. Backend route handlers remain thin and delegate data access/orchestration to services, while the actual mentor agent lives under `backend/src/agents/`. The web app adds a reusable ChatKit React launcher to the existing shell instead of a standalone page, and the browser calls the FastAPI backend directly.

**Indexing Decision**: Course content is chunked, embedded, and upserted into Neon pgvector only by the content indexing workflow, initially once and again only when R2 source content or chunking/embedding settings change. Each mentor request retrieves from the existing vector database through the agent retrieval tool; chat-time requests must not reprocess the whole course corpus.

## Phase 0: Research

Completed in [research.md](./research.md). Key decisions:

- Treat the mentor as a Phase 2 premium hybrid feature behind premium backend mentor endpoints.
- Use OpenAI Agents SDK with LiteLLM configured for the OpenRouter/Nvidia/Nemotron model in environment variables.
- Use ChatKit Python on the backend and `@openai/chatkit-react` for the floating web chat UI.
- Retrieve grounding evidence from pre-chunked, pre-embedded R2 course content stored in Neon pgvector and exposed to the agent as a function tool; embedding/indexing is a content-update workflow, not a per-chat operation.
- Enforce daily quota in PostgreSQL using a per-user UTC-date usage row.
- Use the full persisted history for display and the latest 20 messages for response context.
