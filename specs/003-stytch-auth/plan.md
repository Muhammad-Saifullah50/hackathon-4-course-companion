# Implementation Plan: Stytch Authentication (Backend API)

**Branch**: `003-stytch-auth` | **Date**: 2026-06-04 | **Spec**: [spec.md](spec.md)
**Scope**: Backend only — MCP server auth deferred to the ChatGPT MCP feature

## Summary

Add Stytch JWT authentication to the FastAPI backend. The backend gains: a reusable `get_current_user` FastAPI dependency that verifies Stytch JWTs locally via cached JWKS, a `users` table with first-login provisioning, and a `GET /users/me` endpoint. No MCP server work. Zero LLM calls.

## Design Decisions

### D1 — Stytch Client as Application-Scoped Singleton

`stytch.Client` is initialized once in the FastAPI lifespan and reused across all requests, so JWKS is fetched once and cached. Reinitializing per-request would refetch JWKS on every call.

```python
# src/core/auth.py
_stytch_client: Client | None = None

def get_stytch_client() -> Client:
    global _stytch_client
    if _stytch_client is None:
        _stytch_client = Client(
            project_id=settings.stytch_project_id,
            secret=settings.stytch_secret,
        )
    return _stytch_client
```

The `lifespan` in `main.py` calls `get_stytch_client()` at startup so JWKS is warm before the first request.

### D2 — Sync Dependency (Acceptable for Local JWT Verification)

The `stytch` SDK is synchronous. FastAPI runs sync `Depends()` functions in a thread pool automatically. Because `authenticate_jwt()` with a warm JWKS cache is a pure in-process crypto operation (~1ms, no network call), thread pool overhead is negligible in practice. The background JWKS refresh (every 15 min by the SDK) is the only case that touches the network, and that runs on a separate scheduler thread, not the request thread pool.

```python
# src/core/auth.py
def get_current_user(
    credentials: Annotated[HTTPAuthorizationCredentials, Security(HTTPBearer())],
) -> AuthenticatedUser:
    try:
        resp = get_stytch_client().sessions.authenticate_jwt(
            session_jwt=credentials.credentials
        )
        return AuthenticatedUser(
            user_id=resp.session.user_id,
            email=_extract_email(resp),
        )
    except HTTPException:
        raise
    except Exception:
        raise HTTPException(status_code=401, detail="Invalid or expired token")


def _extract_email(resp: Any) -> str:
    """Extract email from Stytch response — location varies by auth method."""
    # Primary: resp.user.emails (populated for all auth methods)
    emails = getattr(getattr(resp, "user", None), "emails", None)
    if emails:
        return emails[0].email
    raise HTTPException(status_code=401, detail="Email claim missing from token")
```

**Why `resp.session.attributes.email` is not used**: Stytch only populates session attributes for sessions created with certain auth methods. `resp.user.emails` is the authoritative source and is always present when the SDK verifies a valid session.
```

### D3 — Stytch `user_id` as Primary Key

The Stytch `user_id` (stable across login methods) is the `User.id` primary key. No separate UUID column — eliminates a join and a source of truth conflict.

### D4 — First-Login Provisioning via `ON CONFLICT DO NOTHING`

`GET /users/me` calls `UserService.get_or_create()` on every request — idempotent. PostgreSQL `ON CONFLICT DO NOTHING` means no error on repeat calls.

```python
# src/services/users.py
async def get_or_create(self, user_id: str, email: str, db: AsyncSession) -> User:
    stmt = pg_insert(User).values(id=user_id, email=email)
    stmt = stmt.on_conflict_do_nothing(index_elements=["id"])
    await db.execute(stmt)
    await db.commit()
    result = await db.execute(select(User).where(User.id == user_id))
    return result.scalar_one()
```

### D5 — `CurrentUserDep` Annotation

All future protected routes consume a single typed annotation — no boilerplate duplication:

```python
# src/core/dependencies.py
CurrentUserDep = Annotated[AuthenticatedUser, Depends(get_current_user)]
DbSessionDep = Annotated[AsyncSession, Depends(get_db)]
```

## Resolved Questions

| # | Question | Answer |
|---|----------|--------|
| 1 | Exact Stytch JWT `aud` claim format? | Set to the **Project ID** by Stytch. Validated automatically by the SDK using the `project_id` passed to `Client()` — no manual audience check needed. |
| 2 | Where is email in the Stytch SDK response? | `resp.user.emails[0].email` — always present when the SDK verifies a valid session. `resp.session.attributes.email` is **not** reliable across all auth methods (magic link, OAuth, passwords). See D2 for the safe extraction pattern. |
